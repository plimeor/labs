# 计划：Anchor V1

创建日期：2026-05-24

## 背景

Anchor 是一个 note-first, agent-safe knowledge app。V1 的目标不是一次性实现完整知识管理平台，而是打穿本地优先笔记本体和受控 agent 写入边界。

核心技术栈固定为：

- Tauri v2
- Rust
- SolidJS
- Tailwind CSS
- Tiptap
- TanStack Solid Router
- TanStack Solid Query
- TanStack Solid Virtual
- Vite v8
- SQLite

产品边界来自 `docs/ideas/2026-05-23-note-first-agent-safe-knowledge-app.md`：

- Markdown 文件是笔记正文的 source of truth。
- SQLite 只承担每次打开应用重建的会话投影和缓存。
- 用户笔记只有 `Journal` 和普通 `Note`。
- Object Type、Properties、Graph 和 Agents 是增量增强层。
- Agent 写入必须经过本地操作边界、diff、审批和 operation record。

## V1 目标

Anchor V1 交付一个可本地使用的桌面应用：

- 用户可以创建或打开一个本地 vault。
- 用户可以创建、打开、编辑当天 `Journal` 和普通 `Note`。
- 用户通过基于 Tiptap 的 Bear-like 单 surface Markdown 编辑器写作；正文变更 auto-save，持久化边界仍是 Markdown。
- 系统每次打开应用时从 Markdown 重建 SQLite 投影。
- 用户可以搜索笔记，查看链接、反链、未链接提及和当前笔记局部图谱。
- 用户可以给普通 Note 设置 Object Type 和 Properties。
- 用户可以通过统一 task surface 让 agent 基于明确 scope 产生 `Draft`、`Reference`、`Proposal` 或 `Proposed Change`。
- 任何 agent 造成的文件、metadata 或 Graph 改动都通过 `Notes Operation Core` 预览、审批、应用和记录。

## Markdown 编辑体验目标

V1 编辑器不是分屏 preview、source mode 或把 Tiptap JSON 作为 source 的 rich-text 编辑器。目标是一块 Bear-like 主写作 surface：

- 用户在同一个 editor 里写作，heading、list、checkbox、quote、code fence、table 和 Markdown link 即时呈现为接近阅读状态的排版。
- 用户可以通过 Markdown shortcuts、键盘输入和弱工具条创建或修改 `#`、`-`、`1.`、`>`、`` ``` ``、`[[link]]`、`#tag`、inline code、bold / italic、checkbox 等语法；编辑器不要求用户切到独立源码视图。
- `[[link]]`、`#tag`、Markdown link、checkbox、heading、list、blockquote、code fence 和 table 至少有明确的 V1 支持或保存阻断规则。
- 默认不展示 frontmatter 原文；metadata 通过 panel 编辑。
- 正文没有显式 Save 按钮；auto-save 前必须能判断当前内容是否能安全序列化为 Markdown；不能无损 round-trip 的内容必须阻止静默保存并保留本地编辑状态。
- 所有最终写入都提交 Markdown body、frontmatter metadata patch 和 base revision 给 `Notes Operation Core`。

## 实施顺序

1. Desktop shell
   - 先建立 Tauri v2 + SolidJS 的可运行外壳、路由和 Tauri command bridge。
2. Vault and index
   - 建立本地 vault、Markdown 文件发现、打开应用时的 SQLite projection 重建和基础查询。
3. Operation Core
   - 把读写从 UI 与文件细节中抽出，形成所有入口复用的本地语义操作层。
4. Note model and editor
   - 先验证 Tiptap 能否满足 Bear-like 单 surface、auto-save 与 round-trip 矩阵，再在 Operation Core 上实现 Journal、普通 Note、metadata 和 Markdown 编辑。
5. Relations, search, graph
   - 扩展索引和 UI，补齐链接、反链、搜索和局部图谱。
6. Agent-safe tasks
   - 在已有 Operation Core 和 approval surface 上接入 agent task，而不是让 agent 直接读写文件。
7. Investor readiness gate
   - 用原始 idea 的产品承诺做 traceability matrix，确认 demo flow、packaged artifact、offline evidence、agent evidence 和 data-loss evidence 都通过。

## V1 数据边界

Markdown 和 sidecar 文件承担可迁移资产：

- Note 正文。
- `Journal` / `Note` / `Reference` / `Proposal` 的 metadata。
- Object Type、Properties、Agent Command、Automations 和 Agent Spec 配置。
- Operation records。

SQLite 承担打开应用时重建的会话投影：

- note 列表和路径映射。
- links、backlinks、tags、properties 和 graph edges 的索引。

SQLite 没有迁移路径、版本兼容承诺或持久脏状态。应用打开 vault 时删除或覆盖旧 `index.sqlite`，按当前应用版本重新扫描 Markdown、config 和 operation records。正文搜索不进入 SQLite index；V1 由 Rust 直接扫描 vault 内 Markdown 文件，SQLite 只提供路径和 metadata filter 的候选集。

不进入 Markdown 资产：

- OAuth token。
- API key。
- 本地会话凭据。
- provider 私有缓存。

## V1 非目标

不包含：

- 云同步、账号系统或多设备冲突解决。
- Web 或移动端交付。
- 完整多 agent 编排系统。
- 通用 agent marketplace。
- 外部 agent 直接读写 Markdown 作为官方集成路径。
- Tana-like outliner 作为主交互模型。
- 强制所有笔记进入类型系统。
- 全库复杂大图布局。
- 将 agent 推断关系直接写入 Core Graph。
- 复杂 review queue 作为默认工作流。

## 跨模块不变量

- UI、CLI、ACP adapter 和内置 provider 都不能绕过 `Notes Operation Core` 写文件。
- 创建普通 Note 时必须按用户本地时间关联到对应 `Journal`。
- 每次打开应用时，基础笔记、metadata、链接索引和局部图谱从 vault 重建；正文搜索仍能直接扫描 Markdown 文件。
- Tiptap 的 ProseMirror document state 不作为持久化 source；保存时必须回写 Markdown。
- Editor 不得静默改写无法无损 round-trip 的 Markdown；不支持结构必须保留原文并阻断 auto-save。
- Markdown relation parser 必须记录 source span，至少区分正文、frontmatter、code fence、inline code 和 link text 的语境。
- `Reference` 的可信度来自明确 sources，不来自 agent 本身。
- `Proposal` 进入用户笔记或 Core Graph 前需要用户确认。
- Graph edge 必须记录来源、状态和是否进入 Core Graph。
- Mutating operation 必须携带 base revision 或等价版本检查。
- Agent mutating operation 必须形成 diff preview 或可撤销 operation record。

## 总体验收标准

- 用户能在桌面端创建或打开 vault。
- 用户能创建当天 Journal、创建普通 Note、用 Tiptap 编辑内容，正文 auto-save，并在重新打开后从 Markdown 恢复。
- 编辑器默认是 Bear-like single writing surface：同一 surface 内直接写作、渲染常用 Markdown 结构、无 source mode、无显式正文 Save。
- Markdown round-trip smoke 覆盖 frontmatter、headings、lists、checkbox、blockquote、code fence、table、`[[link]]`、Markdown link、`#tag`、property link 和不支持语法；不支持语法不能被静默改写。
- 打开应用后，系统能从 Markdown、config 和 operation records 重建 note list、links、backlinks 和 Local Graph；正文搜索不依赖 SQLite 正文索引。
- 用户能搜索笔记正文和 metadata。
- 用户能查看当前笔记的一跳或两跳局部图谱。
- 用户能设置 Object Type 和 Properties，并让 property link 进入关系索引。
- Agent task 能在明确 scope 下读取笔记，生成 `Draft`、`Reference`、`Proposal` 或 `Proposed Change`。
- `Proposed Change` 展示 target notes、diff、Graph impact、provenance、mode 和 approval state。
- 接受 `Proposed Change` 后，写入通过 `Notes Operation Core` 完成，并记录 operation record。
- 投资人 demo 使用 packaged desktop artifact、golden demo vault 和固定 demo script，不依赖开发者手动修数据库或修改文件。
- V1 evidence bundle 能追踪每个原始产品承诺对应的功能、验证命令、manual evidence、已知限制和 demo 风险。

## 验证方式

实现阶段按改动风险选择：

- `bun run check`
- `bun run lint`
- Tauri dev smoke：打开应用、创建 vault、创建 Journal、创建 Note、重启应用。
- Markdown editor smoke：在同一 editor surface 中输入 heading、list、checkbox、quote、code fence、table、`[[link]]` 和 `#tag`，确认正文 auto-save 后 Markdown 文件可读。
- Markdown round-trip fixture：对支持语法要求 Markdown 输出无未批准 diff；对不支持语法要求保存阻断。
- Index smoke：打开应用时重建 SQLite projection，确认 note list、backlinks 和 graph 可恢复；search 直接扫描 Markdown 仍可用。
- Operation smoke：对同一 note 用过期 base revision 写入时得到 conflict。
- Agent smoke：agent 只能产生 preview 或 approved write，不能直接修改 Markdown。
- Investor readiness smoke：按 golden demo script 在无网络和 provider unavailable 状态下验证本体可用，再在一个可用 agent connection 下验证 Reference / Proposal / Proposed Change 流程。

## 停止条件

V1 计划完成时应停在这个状态：

- 六个模块边界清晰，互相依赖顺序明确。
- Markdown source-of-truth、SQLite open-session projection 和 Operation Core write boundary 没有被打破。
- Agent 相关能力只通过 task surface 和 Operation Core 暴露。
- Investor readiness gate 通过，且所有未达成的原始承诺都被明确降级、移出 V1 或暂停给用户确认。
- 未进入 V1 的能力都留在非目标或暂停条件里，不混入默认实施范围。
