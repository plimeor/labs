## 问题陈述

前端团队经常要在多个项目之间判断一份 PRD 会碰到哪些代码、哪些约束、哪些回归点。`code-wiki` 要解决的问题是：让团队能围绕代码提问，用持久代码知识审查 PRD，并在动手前拿到更稳的 coding plan。

## 推荐方向

`code-wiki` 的方向是 multi-project code wiki CLI，服务 codebase Q&A、PRD review 和 coding plan generation。核心对象不是单个仓库的文档集，而是一个可能影响多个项目的 change initiative。每个项目保留自己的 wiki；一次 review run 再把相关 project wikis 拉到一起评估。

第一条有用路径应该是：注册项目，扫描代码生成 project wikis，把 wiki 当成 code context，针对选定项目集合审查 PRD，然后输出 missing requirements、affected modules、project-level code plans、integration risks、regression scope 和 open questions。这条路径直接覆盖三件事：问代码、审 PRD、产出 coding plan。

Upstream CodeWiki 在 code scanning、component extraction、module clustering、incremental update 和 markdown output 上值得参考，但不应该照搬它的产品边界。这里 repository documentation 只是输入层，不是最终目的。真正要验证的是 PRD-to-plan loop，尤其是一份 PRD 同时影响多个 frontend applications 和 monorepo packages 的场景。

## 需要验证的关键假设

- [ ] 自动扫描生成的 module 和 flow knowledge 足够支撑 PRD review。验证方式：扫描一个独立 frontend app 和一个 monorepo package，再把生成的 wiki pages 跟已知项目结构对照。
- [ ] 团队愿意修正重要 wiki 错误。验证方式：跑两次真实 PRD review，看 reviewers 是否愿意补 corrections 或 crystallized notes。
- [ ] wiki 不完整时，PRD review 仍有价值。验证方式：检查 partial wiki 是否至少指出一个 meaningful missing requirement、integration risk 或 regression scope item。
- [ ] 跨项目集成风险足够常见，值得一等支持。验证方式：选几份近期影响多个项目的 PRDs，对比生成的 integration plan 和实际 implementation path。

## MVP 范围

MVP 先支持一个包含多个 registered projects 的 workspace：

```bash
code-wiki projects add web-app --path /path/to/web-app
code-wiki projects add platform --path /path/to/monorepo
code-wiki scan web-app
code-wiki scan platform
code-wiki review prd.md --projects web-app,platform
code-wiki correct web-app correction.md
```

每个 project 有自己的 file-backed wiki：

- `overview.md`：project purpose、boundaries、entry points 和 major subsystems
- `index.md`：给 agents 和 humans 使用的 routing index
- `log.md`：scan、correction 和 crystallization events 的 append-only maintenance log
- `modules/*.md`：module responsibilities、key files、dependencies、common change points 和 regression hints
- `flows/*.md`：可以推断或人工修正的重要业务/技术 flows
- `decisions/*.md`：human-confirmed constraints、historical decisions 和 recurring implementation rules
- `metadata.json`：repo path、commit、include/exclude patterns 和 module tree references 等 generated scan metadata

`review` 输出四层报告：

- Cross-project summary：affected projects、shared contracts、integration dependencies、release ordering 和 rollback boundaries
- Missing requirements：global gaps 加 project-specific omissions
- Project plans：每个 project 的 affected modules、implementation steps、local risks、regression scope 和 verification notes
- Integration plan：API/data contracts、coordination points、joint testing、rollout order 和 open questions

`correct` 是 correction 和 crystallization path。它把 human-confirmed fixes 写回相关 project wiki，避免把自动扫描输出误当成最终事实。

## 不做

- 完整 CodeWiki 或 DeepWiki clone：第一版不做 HTML viewer、GitHub Pages publishing、rich diagrams，也不追求 comprehensive repository documentation。
- 全自动 project selection：可以提供 project selection report，但 review 必须要求明确的 `--projects` 列表或明确的 `--all-projects` 选择。
- 自动改写 PRD：工具只审查 PRD 并产出 code plans，不静默编辑 requirements。
- Cross-project knowledge graph：MVP 中 project wikis 保持分离。跨项目推理只发生在 review runs 内部。
- 隐藏数据库作为 source of truth：markdown 和小型 metadata files 仍然是 durable state。search indexes 或 embeddings 以后只能作为 derived cache。
- 复杂团队 workflow：第一版不做 permissions、web app、approval UI 或 collaboration system。Git-reviewed markdown 已经够用。

## 开放问题

- workspace state 应放在 repo-local `.code-wiki/`、user-level `~/.code-wiki/`，还是两者都支持？
- generated project wikis 应放在每个 target repository 内、`labs` workspace 内，还是独立 team knowledge repo？
- 第一版接受哪些 PRD formats：markdown only、本地 docs 转成 markdown，还是从内部系统导出的 links？
- 第一版 implementation 应面向哪个 model/runtime：direct OpenAI-compatible API、local Codex-driven workflow，还是一开始就 provider abstraction？
- Upstream CodeWiki 应 port、wrap，还是只作为设计参考研究？
