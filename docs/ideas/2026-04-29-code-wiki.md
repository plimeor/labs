# Code Wiki

Created: 2026-04-29
Status: Idea snapshot

This file captures the initial product direction and MVP boundary. Do not keep it synchronized with implementation. When `code-wiki` becomes active work, promote the relevant parts into a living spec under `docs/specs/` and, if needed, an implementation plan under `docs/plan/`.

## Problem Statement

How might we help a six-person frontend team review PRDs against durable code knowledge across multiple projects, so missing requirements are caught earlier and implementation plans become more stable?

## Recommended Direction

Build `code-wiki` as a PRD review-first, multi-project code wiki CLI. The core object is not a repository document set; it is a change initiative that may affect several projects. Each project has its own wiki, but a review run can combine several project wikis into one cross-project assessment.

The first useful workflow is: register projects, scan code into project wikis, review a PRD against a selected project set, and produce a report with missing requirements, affected modules, project-level code plans, integration risks, regression scope, and open questions. This makes the tool directly serve the two success criteria: fewer missed requirements and steadier plans.

Upstream CodeWiki is useful prior art for code scanning, component extraction, module clustering, incremental update, and markdown output. Its product boundary should not be copied wholesale: repository documentation is an input layer here, not the primary deliverable. The differentiator is the PRD-to-plan loop, especially when one PRD spans multiple frontend applications and monorepo packages.

## Key Assumptions to Validate

- [ ] Automatic scanning can produce enough accurate module and flow knowledge to support PRD review. Test by scanning one independent frontend app and one monorepo package, then manually compare generated wiki pages against known project structure.
- [ ] The team will correct important wiki mistakes after review runs. Test by running two real PRD reviews and checking whether reviewers are willing to add corrections or crystallized notes.
- [ ] PRD review value appears before the wiki is complete. Test whether a partial wiki can still surface at least one meaningful missing requirement, integration risk, or regression scope item.
- [ ] Cross-project integration risks are common enough to justify first-class support. Test on recent PRDs that affected more than one project and compare the generated integration plan against the actual implementation path.

## MVP Scope

The MVP should support a workspace with multiple registered projects:

```bash
code-wiki projects add web-app --path /path/to/web-app
code-wiki projects add platform --path /path/to/monorepo
code-wiki scan web-app
code-wiki scan platform
code-wiki review prd.md --projects web-app,platform
code-wiki correct web-app correction.md
```

Each project owns a file-backed wiki:

- `overview.md`: project purpose, boundaries, entry points, and major subsystems
- `index.md`: routing index for agents and humans
- `log.md`: append-only maintenance log for scan, correction, and crystallization events
- `modules/*.md`: module responsibilities, key files, dependencies, common change points, and regression hints
- `flows/*.md`: important business or technical flows when they can be inferred or manually corrected
- `decisions/*.md`: human-confirmed constraints, historical decisions, and recurring implementation rules
- `metadata.json`: generated scan metadata such as repo path, commit, include/exclude patterns, and module tree references

`review` should output one report with four layers:

- Cross-project summary: affected projects, shared contracts, integration dependencies, release ordering, and rollback boundaries
- Missing requirements: global gaps plus project-specific omissions
- Project plans: affected modules, implementation steps, local risks, regression scope, and verification notes for each project
- Integration plan: API/data contracts, coordination points, joint testing, rollout order, and open questions

`correct` is the correction and crystallization path. It should write human-confirmed fixes back into the relevant project wiki instead of treating automatic scan output as final truth.

## Not Doing

- Full CodeWiki or DeepWiki clone: HTML viewer, GitHub Pages publishing, rich diagrams, and comprehensive repository documentation are not first-version goals.
- Fully automatic project selection: the first version may provide a project selection report, but review should require an explicit `--projects` list or an explicit `--all-projects` choice.
- Automatic PRD rewriting: the tool reviews PRDs and produces code plans; it does not silently edit requirements.
- Cross-project knowledge graph: project wikis remain separate in the MVP. Cross-project reasoning happens inside review runs.
- Hidden database as source of truth: markdown and small metadata files remain the durable state. Search indexes or embeddings may be added later as derived cache only.
- Complex team workflow: no permissions, web app, approval UI, or collaboration system in the first version. Git-reviewed markdown is enough for the initial six-person team.

## Open Questions

- Where should workspace state live: repo-local `.code-wiki/`, user-level `~/.code-wiki/`, or both?
- Should generated project wikis live inside each target repository, inside the `labs` workspace, or in a separate team knowledge repo?
- What PRD formats should be accepted first: markdown only, local docs copied to markdown, or links exported from an internal system?
- Which model/runtime should the first implementation target: direct OpenAI-compatible API, local Codex-driven workflow, or provider abstraction from day one?
- How much of upstream CodeWiki should be ported, wrapped, or only studied as design reference?
